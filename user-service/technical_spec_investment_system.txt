Техническое задание
Система управления инвестиционными проектами
1. Общая информация
Документ описывает техническое задание на разработку системы управления инвестиционными проектами.
Цель документа — определить функциональные возможности системы, архитектуру, используемые технологии и структуру данных.
2. Назначение системы
Система управления инвестиционными проектами предназначена для автоматизации процессов подачи, рассмотрения, оценки и хранения инвестиционных проектов внутри организации.
Система должна решать следующие задачи:
•	Регистрация и управление пользователями системы
•	Подача инвестиционных проектов
•	Хранение и управление нормативно‑справочной информацией (НСИ)
•	Оценка и ранжирование проектов по заданным характеристикам
•	Хранение документов проекта
•	Формирование отчётов и выгрузок
•	Предоставление данных для аналитической системы
3. Пользователи системы
В системе предусмотрены следующие роли пользователей:
•	Заявитель — пользователь, который создает и подает инвестиционные проекты.
•	Инвестор — пользователь, который рассматривает и оценивает проекты.
•	Администратор — пользователь, который управляет системой, справочниками и пользователями.
4. Основные функции системы
4.1 Управление пользователями
•	Создание пользователей администратором
•	Назначение ролей
•	Блокировка и разблокировка пользователей
•	Сброс пароля
•	Авторизация пользователей
4.2 Управление инвестиционными проектами
•	Создание проекта
•	Редактирование проекта
•	Отправка проекта на рассмотрение
•	Просмотр списка проектов
•	Изменение статуса проекта
•	Добавление этапов проекта
•	Хранение характеристик проекта
•	Хранение истории статусов
4.3 Управление НСИ
•	Справочник направлений инвестиций
•	Справочник характеристик проектов
•	Настройка весов характеристик для ранжирования
•	Настройка диапазонов значений характеристик
4.4 Работа с документами
•	Загрузка ТЭО проекта
•	Хранение файлов в файловой системе сервера
•	Формирование документа Word с информацией о проекте
5. Архитектура системы
Система реализуется с использованием микросервисной архитектуры. Каждый сервис отвечает за отдельную область бизнес‑логики.
Основные сервисы системы:
•	User Service — управление пользователями и ролями
•	NSI Service — управление нормативно‑справочной информацией
•	Project Service — управление инвестиционными проектами
•	Analytics Service — аналитика и отчеты (разрабатывается отдельно)
•	Gateway — единая точка входа для клиентского приложения
6. Используемые технологии
•	Java (Spring Boot) — backend сервисы
•	React — пользовательский интерфейс
•	PostgreSQL — система управления базой данных
•	gRPC — взаимодействие между микросервисами
•	JWT — авторизация пользователей
•	bcrypt — хеширование паролей
•	Apache POI — генерация документов Word
7. Структура базы данных
База данных PostgreSQL используется всеми сервисами, но каждый сервис работает со своей схемой.
Основные таблицы:
•	users — пользователи системы
•	roles — роли пользователей
•	projects — инвестиционные проекты
•	project_stages — этапы проекта
•	project_characteristics — характеристики проекта
•	project_status_history — история изменения статусов
•	directions — направления инвестиций
•	scoring_rules — правила расчета рейтинга проектов
8. Характеристики проектов
Характеристики проектов хранятся в универсальной структуре, что позволяет добавлять новые показатели без изменения структуры базы данных.
•	ROI — рентабельность инвестиций
•	IRR — внутренняя норма доходности
•	NPV — чистая приведенная стоимость
•	PI — индекс прибыльности
9. Ранжирование проектов
Для оценки проектов используется балльная система. Каждая характеристика имеет определенный вес.
Формула расчета рейтинга:
Score = (характеристика × вес) + ...
10. Хранение файлов
Документы проекта (ТЭО и другие файлы) хранятся в файловой системе сервера. В базе данных хранится только путь к файлу.
11. Архив проектов
Проекты могут быть переведены в статус ARCHIVED. Такие проекты не отображаются в основном списке, но сохраняются в базе данных.
12. Безопасность системы
•	Авторизация через JWT
•	Хеширование паролей с использованием bcrypt
•	Разграничение прав доступа по ролям

6. Service слой
________________________________________
AuthService
@Service
@RequiredArgsConstructor
public class AuthService {

    private final UserRepository userRepository;
    private final JwtService jwtService;
    private final PasswordEncoder passwordEncoder;

    public String login(LoginRequest request){

        User user = userRepository
                .findByLogin(request.getLogin())
                .orElseThrow();

        if(!passwordEncoder.matches(
                request.getPassword(),
                user.getPasswordHash()
        )){
            throw new RuntimeException("Invalid password");
        }

        return jwtService.generateToken(user);
    }

}

7. JWT сервис
JwtService
@Service
public class JwtService {

    private final String SECRET = "very_secret_key";

    public String generateToken(User user){

        return Jwts.builder()
                .setSubject(user.getLogin())
                .claim("userId", user.getId())
                .setIssuedAt(new Date())
                .setExpiration(new Date(
                        System.currentTimeMillis() + 86400000
                ))
                .signWith(SignatureAlgorithm.HS256, SECRET)
                .compact();
    }

}

8. SecurityConfig
@Configuration
@EnableWebSecurity
public class SecurityConfig {

@Bean
public PasswordEncoder passwordEncoder(){
    return new BCryptPasswordEncoder();
}

@Bean
public SecurityFilterChain filterChain(
        HttpSecurity http
) throws Exception {

    http
        .csrf().disable()
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/auth/**").permitAll()
            .anyRequest().authenticated()
        );

    return http.build();
}

}
